# 项目演进脉络

这份文档不试图重复所有细节，而是把项目一路走到现在的关键转向压出来，方便下一位 agent 快速看懂“为什么仓库会长成现在这样”。

## 1. 起点：先做 skill，用产物倒逼理解方法

项目最早的出发点，不只是做一个六问法工具，而是：

- 借着“做一个 skill”来倒逼自己理解六问法
- 顺便保留一个以后自己可能会反复使用的工具

真正的前史其实发生在当前仓库建立之前：

- 在 Claude Code 里，用户先拿原始 Word 文档直接做最早的六问法 skill 原型
- 随后在 Codex 里又做了一轮迁移、安装、测试与卸载

这些前史现在已经补进 `evals/sessions/raw/`，因为它们解释了项目最初为什么会从“直接做 skill”开始。

后面仓库才逐渐被按 `yc_think` 这个教练型元 skill 来组织起来。

## 2. 第一阶段：围绕 skill 本体做设计与测试

这一阶段的重点是：

- 明确 `yc_think` 的定位
- 让它默认中文、默认互动
- 把第 1-3 问改成先问后导
- 把 `0.5Why` 的显式边界收回到第 4、5 问

这就是后来的 `v0.2`。

这一阶段的价值仍然很高，因为它沉淀了：

- 六问法的操作骨架
- 一部分真实测试反馈
- 哪些地方最容易被 LLM 做偏

## 3. 第二阶段：老师反馈与原始 source 校准，开始暴露方法边界问题

随着原始 source 与测试结果一起被重新阅读，关键偏差开始暴露出来：

- 第三问和第四问容易混在一起
- 第三问不应该过早找“核心差距”
- 第三问只需要把目标、现状、差距列清
- 第四问才真正处理“目标背后的目的”和“目标是不是方案”

这个阶段的变化很关键，因为项目开始从“怎么把 skill 做得更像六问法”转向“我自己到底有没有真正理解六问法”。

## 4. 第三阶段：先训练成教练，再回头设计 skill

这一步是一个真正的方向转向。

用户明确提出：

- 今天先不急着继续升级 skill
- 先把自己训练成一个更理解六问法的人
- 最好先能达到“六问法教练”的理解水平

于是仓库新增了 [../六问法教练训练](../六问法教练训练) 这组材料，把“方法理解”与“skill 设计”拆开。

## 5. 第四阶段：新路加入，六问法不再只是问题界定工具

结合 `chatgpt.com-六问法与生命观融合` 这份原始材料，项目又多了一层变化：

- 六问法不只被看作服务他人解决问题的工具
- 它也开始被看作一种觉察训练工具
- 更准确的说法不是“替代六问法”，而是给六问法增加一个新维度

于是形成了一个更稳的理解：

- 服务模式：守住六问法骨架，帮助界定问题、支持行动
- 觉察模式：看清问题、目标和差距是如何被制造出来的

但这一步同时带来新的风险：

- 如果不分模式，第三问与第四问会更容易混
- 如果觉察过度灵性化，会冲散六问法的骨架

## 6. 第五阶段：真实练习暴露出用户自己的使用风格

在 `019d22a8` 与 `019d2371` 这两个 session 里，用户逐渐看清了自己更真实的使用方式：

- 自己不太适合完全一板一眼地走流程
- 更在意背后的目的与驱动力
- 一旦目的和目标足够清楚，就更倾向于先行动
- 借助 AI 去推进、试错、调整，比继续长时间分析更自然

这里最重要的收敛是：

`用户真正想训练的，不是怎样死守流程，而是怎样借助方法更快抓住真实目的，并让行动更顺。`

## 7. 当前阶段：仓库被重新定位为“下一位 agent 的素材库”

到当前这一步，项目的主要价值已经不只是“skill 本体”了。

新的优先级变成：

- 把原始 source 和原始 session 留全
- 把项目过程中的关键转向留清楚
- 让下一位 agent 能快速吸收这些材料
- 让下一位 agent 更像“思维教练顾问”，而不是“只会继续改 skill 的执行器”

## 8. 对下一位 agent 最有价值的启发

这一路最值得继承的，不是某个固定 prompt，而是下面这些启发：

1. 对这个用户来说，做产品常常只是为了倒逼理解。
2. 当方法理解不够扎实时，继续迭代 skill 会越做越乱。
3. 六问法对他有帮助，但不能被僵化使用。
4. 目标层级要分清，不然第四问很容易把问题提得过大。
5. 用户现在更需要的是一个能一起挖意图、一起行动的 agent，而不是一个只会守模板的 agent。
