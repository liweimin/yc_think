# yc_think v0.1 升级分析

## 1. 升级背景

这次升级来自第一次真实测试反馈。

测试 prompt：

- Use yc_think. 我总是拖延本月学习任务，但我也说不清真正的问题是什么。请按这个方法帮我分析。

用户反馈的两个关键问题：

1. 最严重的问题是，模型自己一路回答完了，没有和用户互动走完整个流程。
2. 每一问最好默认使用中文。

## 2. v0.0 的主要问题

### 问题 1：默认行为太偏“自动完成”

v0.0 的设计虽然强调了六问法结构，但没有把“互动”写成硬约束。

结果是模型会自然选择：

- 用已有信息自行推断
- 尽快补全整个分析链
- 直接给出完整结论

这在一般分析任务里看起来像“主动”，但对 `yc_think` 这种教练型 skill 来说是错误行为。

### 问题 2：输出模板默认鼓励“终局答案”

v0.0 的输出模板重点是完整结果：

- 原始问题
- 支撑层
- 轮次过程
- 最终结果

这会让模型倾向于一次把这些都填满，而不是分步互动。

### 问题 3：没有明确区分“互动模式”和“草案模式”

v0.0 没有告诉模型：

- 什么时候该一步步问
- 什么时候才允许先给一版推演草案

于是模型默认走了“草案模式”，但又没有明确标注这是草案。

### 问题 4：中文要求不够硬

虽然项目上下文是中文，但 skill 内没有把“默认中文”写成强规则。

结果是：

- 大部分内容可能是中文
- 但步骤名、问题名、部分术语容易偏英文

这会影响体验的一致性。

## 3. v0.1 的设计目标

v0.1 的目标很明确：

1. 把“互动式教练”变成默认执行方式
2. 把“默认中文”变成明确规则
3. 把“整段自答”限制为特殊模式
4. 让较普通的 LLM 也能稳定照着执行

## 4. v0.1 的核心改动

### 改动 1：把互动写成硬规则

在 `SKILL.md` 中新增了这些明确规则：

- 默认不允许一口气完成全部流程
- 默认一轮只解决一个未完成阶段
- 每一步都要“解释 -> 给方向 -> 提问 -> 等用户”
- 只有用户明确要求“先给草案”时，才允许一版推演

### 改动 2：区分互动模式和草案模式

新增两种模式：

#### 互动模式

- 默认模式
- 一步一步推进
- 每次只推进当前步骤

#### 草案模式

- 只有用户明确要求时才启用
- 必须标明“这是暂定草案”
- 必须标出假设

### 改动 3：输出格式拆成“回合格式”和“最终格式”

v0.1 不再默认让模型直接输出完整终局结构。

现在拆成：

- 回合格式
  - 适配判断
  - 当前步骤
  - 暂定理解
  - 回答方向
  - 请你回答
- 最终格式
  - 在信息足够、或用户要求草案时才使用

### 改动 4：默认中文

v0.1 明确要求：

- 默认中文输出
- 默认中文步骤名
- 默认中文提问

### 改动 5：把“不要自己扩写一整套案例”写清楚

在教练指南中，新增了明确限制：

- 当用户只给一句话时，不要自己扩写成完整案例
- 先分类、解释、给方向、再问下一步

## 5. 为什么这次改动更适合 yc_think

`yc_think` 的本质不是“自动生成一份六问法报告”，而是：

- 陪用户界定问题
- 训练用户思维
- 降低用户回答门槛

如果模型直接一路写完：

- 用户没有参与
- 用户没有被训练
- 用户也无法及时纠正模型的假设

所以，互动不是附加体验，而是核心能力。

## 6. v0.1 重点要验证什么

下一轮测试时，优先看下面几点：

### 6.1 互动性

- 是否默认进入互动模式
- 是否只推进当前一步
- 是否会停下来等待用户回答

### 6.2 中文一致性

- 步骤名是否中文
- 提问是否中文
- 解释是否中文

### 6.3 回答方向是否有帮助

- 是否主动给 2-4 个回答方向
- 是否足够具体
- 是否真的降低了用户回答难度

### 6.4 草案模式是否被正确限制

- 普通情况下是否避免直接整段推演
- 用户明确要求草案时，是否能正确切换

## 7. 版本保留方式

为了保留历史版本：

- `versions/skills/yc-think-0-0` 保留 v0.0
- `skills/yc-think` 作为当前活跃版本 v0.1

## 8. 当前结论

这次升级不是小修小补，而是把 skill 的默认行为从：

- “自己补完”

改成：

- “先互动，再推进；必要时才草案”

这才更符合 `yc_think` 作为教练型元 skill 的定位。
