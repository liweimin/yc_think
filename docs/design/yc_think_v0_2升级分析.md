# yc_think v0.2 升级分析

## 1. 升级来源

- 原始 session: [019d2086-9364-7370-bacd-9cea6da0c510.jsonl](D:\code\0324_2025\evals\sessions\raw\019d2086-9364-7370-bacd-9cea6da0c510.jsonl)

2. 感觉第 3 问就开始出现 `0.5Why` 的味道。
3. 在用户回答前先给回答方向，会干扰用户自己思考；应该先让用户答，只有用户说“给我思路”时再给指引。

## 2. 这轮真实测试说明了什么

### 正面结果

相比 v0.0，v0.1 已经明显改对了一个核心方向：

- 模型不再一口气完成整套分析
- 已经开始按步骤互动

这说明“互动优先”这个大方向是对的。

### 暴露出来的新问题

#### 问题 1：第 3 问的确认方式太像 `0.5Why`

从方法论上说：

- `0.5Why` 应主要用于第 4 问和第 5 问

但在实际对话里，第 3 问后模型用了很多类似：

- “这句差距是不是更接近……”
- “如果只能选一个最核心的差距……”
- “是不是这句……你回答是或不是”

这虽然不是真正显式的 `0.5Why`，但体验上已经很像“验证某一句是否成立”。

这会带来两个问题：

- 用户感觉方法边界被混淆了
- 第 3 问显得过早进入“验证式追问”

#### 问题 2：回答方向给得太早

在大多数轮次里，模型都默认给出了方向选项：

- 没有计划成品
- 标准不清
- 收敛不了
- 边学边飘

这种做法在用户卡住时是有帮助的，但在用户本来能自己组织答案时，会产生副作用：

- 把用户带进模型给的框里
- 压缩用户自己的表达空间
- 干扰我们观察用户原始思路

所以这里的问题不是“给方向不对”，而是“给方向太早”。

## 3. v0.2 的升级目标

v0.2 的目标是：

1. 保留互动性
2. 把 `0.5Why` 的边界收回到第 4、5 问
3. 把“回答方向”从默认动作改成条件动作

## 4. v0.2 的核心改动

### 改动 1：明确禁止在第 1-3 问使用 `0.5Why` 语言

在 `SKILL.md` 和流程文档里写成硬规则：

- 第 1、2、3 问不能显式使用 `0.5Why`
- 第 3 问可以做差距确认和重开判断
- 但不能把这个动作写成 `0.5Why`，也不能用过强的验证式表述

### 改动 2：回答方向不再默认给

v0.2 改成：

- 默认先问
- 默认先看用户怎么答
- 只有在下面情况才给方向：
  - 用户主动要提示
  - 用户明显卡住
  - 用户回答过于模糊，无法继续

### 改动 3：把“先问后导”写成默认回合纪律

原来是：

- 解释
- 给方向
- 提问

现在改成：

- 解释
- 提问
- 必要时再给方向

也就是说，方向不再是固定组成部分，而是按需使用。

## 5. 为什么这次改动更合理

### 对方法更忠实

这让方法边界更清楚：

- 第 1-3 问负责界定
- 第 4-5 问负责深挖和验证

### 对用户体验更好

这让用户先保留自己的原始表达，再在必要时接受引导。

好处是：

- 不会太早被模型框住
- 更容易看到用户真正的思考路径
- skill 更像教练，不像带答案的问卷

## 6. v0.2 重点验证项

下轮测试优先看：

1. 第 3 问是否不再给人“提前 0.5Why”的感觉
2. 第 1-3 问是否默认先问，不抢着给方向
3. 用户卡住时，方向是否仍然能及时补上
4. 互动感是否保留，没有退回到 v0.0 的“一路自答”
