# 第三问与第四问的边界

## 1. 先说结论

第三问和第四问必须分开。

### 第三问只负责

- 得到目标与现状之间的差距
- 判断差距是否存在
- 判断原问题是否成立

### 第四问才负责

- 判断目标背后的目的
- 判断当前目标是不是最终目标
- 判断当前目标是不是临时方案

如果把这两问混在一起，分析会乱掉。

## 2. 原始资料是怎么说的

从原始 `docx` 和示例材料看，源头逻辑是很清楚的。

### 原始方法里的信号

- 第三问写的是“差距是什么”
- 原文明确说“通过第三问，确定差距后”，还要继续问第四到第六问，去判断这个差距是不是真正的问题
- 第四问写的是“目标背后的目的是什么”

也就是说，原始方法本身已经把两问分开了：

- 第三问先把差距定出来
- 第四问再判断这个目标和差距值不值得按原样去解决

### 示例材料里的信号

在“优化业务流程”示例中：

- 第三问只写出差距
- 第四问才进入“为什么要这个目标”和“这个目标是不是唯一办法”

这和老师的反馈是一致的。

## 3. 老师反馈到底在纠正什么

老师给的两个反馈，可以翻译成下面两条正式规则。

### 规则 1：第三问不要提前做第四问

老师反馈：

- 第三问跟第四问合并在一起了
- 差距内容，只需要得到目标-现状，清楚差距是什么即可
- 对内容和目的的合适性判断，属于第四问

这说明：

- 第三问不能提前去判断“这个目标该不该要”
- 第三问不能提前去判断“这是不是更深层真正问题”
- 第三问更不能提前开始“这个回答一定意味着那个目标吗”这种验证

### 规则 2：第三问不要急着找唯一核心差距

老师反馈：

- 在第三问，不用找核心差距
- 识别出来每个维度的差距都要进行分析

这说明：

- 如果一个问题天然有多个并列维度，就要先把这些差距列清楚
- 不要在第三问过早做“如果只能选一个最核心的差距”
- “收缩成一个核心矛盾”不是第三问的默认动作

## 4. 第三问到底该怎么做

第三问最稳的做法是：

1. 先确认目标和现状是否同维度
2. 再把每个维度上的差距写出来
3. 检查这些差距是否构成原问题

第三问的典型提问应更像：

- 目标和现状放在同一维度看，分别是什么？
- 现在到底差在哪几个地方？
- 每个维度上，目标和现状分别差多少？
- 这样写出来后，原问题还成立吗？

第三问不该优先问成：

- 如果只能选一个最核心差距，会是哪一个？
- 真正的差距更像落在哪一句？
- 这句差距是不是更接近真正问题？

这些问法太容易把第三问带进第四问甚至第六问。

## 5. 第四问到底该怎么做

第四问最重要的任务，是拆穿“把方案当目标”的情况。

第四问应当问：

- 你为什么一定想要这个目标？
- 这个目标背后真正想解决的是什么？
- 除了这个目标，还有没有别的办法达到你真正想要的结果？

这一问才适合进入：

- 5Why
- 0.5Why
- 目标是否只是临时方案
- 是否要重开

## 6. 当前设计的偏差，具体偏在哪

结合当前仓库里的文档，我的判断是：

### 整体骨架没有偏

当前文档大方向是对的：

- 0.5Why 已经被明确收回到第四问和第五问
- 也明确写了第三问和第四问都可能触发重开

这说明总骨架没有走偏。

### 真正的偏差在执行层

偏差主要落在两点：

1. 第三问的引导语言还是有点太强，容易出现“提前验证感”
2. 第三问有时会被带成“帮用户找一个最核心差距”，而不是先把差距列清

所以，这不是“六问法框架错了”，而是：

**第三问的动作边界还不够收。**

## 7. 之后改 skill 时，第三问建议改成什么

后续如果回到 skill 迭代，第三问建议改成下面这个方向：

### 第三问默认动作

- 先对齐目标与现状的维度
- 再列清各维度差距
- 只判断原问题是否成立

### 第三问不要默认做

- 不要找唯一核心差距
- 不要做目的验证
- 不要做强验证式问法
- 不要让用户在第三问就提前回答“真正想解决的是什么”

### 第三问结束时能得到什么

- 差距清单
- 原问题是否成立
- 是否需要基于“原问题不成立”而重开

如果问题成立，就进入第四问。

如果问题不成立，就改写问题并重开。
